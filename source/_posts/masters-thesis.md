---
title: Master's Thesis
---
Domain-oriented test suite generation is an essential technique for making software programs relatively more explainable since it can reason about the governing rules changing program behavior through domain boundaries. It is also crucial for detecting latent faults which do not manifest themselves unless the program executes with specific test data from the feasible domain. However, real-world programs typically contain complex data types such as string, struct and array, and black-box called methods that challenge domain coverage. I have transformed the domain-oriented test suite generation task into a constraint satisfaction problem in my master's thesis entitled *"Development of a Constraint Solver to Determine the Domain for Complex Data Types"*. The state-of-the-art constraint solvers generate only one satisfying assignment and support a limited number of data types. Especially they fail when they encounter black-box called methods which considerably affects the performance of the solvers.

To address the challenges, I have proposed a technique for constraint solving, called *DomainSolver*, which starts with simplifying the constraint using some testability transformations and splits the result into some sub-constraints each of which contains some expressions with a similar data type. As such splitting guarantees that there is no common variable between the sub-constraints, we can independently solve each sub-constraint by an appropriate strategy. *DomainSolver* returns the resultant solutions as a set of feasible sub-domains solving the target constraint, provided that it is satisfiable.

I have implemented the proposed approach, which is developed in C# programing language and uses an underlying compiler as a service, .NET Compiler Platform (Roslyn). I assessed the benefits and costs of Domain-Solver with five well-known constraint solvers, CVC4, S3, Yices2, Z3 Theorem Prover, and Z3str3. The experimental results, based on two industrial benchmarks, SMT-LIB and Kaluza, with 56 constraints confirm that our approach significantly improves the number of supported data types involved in a constraint, and solves all of the instances on which the other solvers fail while performing in an affordable execution time.
